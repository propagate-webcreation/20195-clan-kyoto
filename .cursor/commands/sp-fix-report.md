# /sp-fix-report — SP版修正報告コマンド

スマートフォン版の修正完了後にクライアントへ送る報告メールを自動生成する。
サブコマンドで **REPORT**（報告文生成）と **LINKS**（URL取得のみ）を切り替える。

## 使用方法

```
/sp-fix-report               ← 今回のチャットの修正内容で報告文を生成（デフォルト）
/sp-fix-report 4/1~          ← 4/1以降の修正内容で報告文を生成
/sp-fix-report 3/28~4/5      ← 3/28〜4/5の範囲で報告文を生成
/sp-fix-report links         ← LINKS のみ（URL取得だけ）
```

---

## LINKS — 制作進行シートからURL取得

制作進行シート（GAS API）から、現在のリポジトリに対応する **VercelリンクとアノテーターURL** を取得する。

### Step 1. リポジトリ名の自動取得

```bash
repo_name=$(basename $(git remote get-url origin) .git)
```

### Step 2. GAS APIを呼び出す

```bash
curl -sL "https://script.google.com/macros/s/AKfycbwLQ9BrxjcMpp2cUrk7AeEddqRBrDasPNOXYwIhZOtpCzwo4D2s88pRnHsEKSzEQP4i/exec?repo=$repo_name"
```

### Step 3. レスポンスの解析と出力

#### 1件のみの場合

```
Vercel URL    : https://xxx.vercel.app/
アノテーター  : https://site-annotator.vercel.app/share?token=xxxxx
```

#### 複数件ヒットした場合

全件を番号付きで即時出力する。

```
# 検索結果: 2件ヒット

---
[1件目] シート: メイン / 行79
Vercel URL    : https://xxx.vercel.app/
アノテーター  : https://site-annotator.vercel.app/share?token=xxxxx

---
[2件目] シート: 完了 / 行12
Vercel URL    : https://xxx.vercel.app/
アノテーター  : https://site-annotator.vercel.app/share?token=yyyyy
```

#### 該当なしの場合

```
「{リポジトリ名}」に該当する行が制作進行シートに見つかりませんでした。
```

### Step 4. アノテーターURLのバリデーション

| 状態 | 対応 |
|------|------|
| `/share` を含む | 正常。そのまま出力 |
| `/share` を含まない（`/admin` 等） | 処理中断。`⚠️ アノテーターのURLが不正です。取得されたURL: {URL} /share を含むURLが設定されていることを確認してください。` |
| null（未登録） | 処理中断。`⚠️ アノテーターのURLが制作進行シートに登録されていません。` |

**`/sp-fix-report links` の場合はここで終了。REPORT には進まない。**

## REPORT — クライアント報告文の自動生成

ユーザーが指定した日付範囲の **Git 履歴・エージェント履歴・対話履歴** を横断的に参照し、修正内容を抽出して報告メールを生成する。
**REPORT は内部で LINKS を先に実行し、URLを自動埋め込みする。**

### STEP 1 — LINKS の実行（URL取得）

上記 LINKS セクションの Step 1〜4 をすべて実行し、`vercel_url` と `annotator_url` を取得する。

- **1件の場合**: そのまま使用
- **複数件の場合**: 全件を表示し、ユーザーに使用する組み合わせを選択してもらう
- **該当なしの場合**: テンプレート内のURL欄は `{VercelのURLを貼る}` `{アノテーターのURLを貼る}` のまま出力する

### STEP 2 — 指定期間の修正内容を抽出

ユーザーが指定した日付範囲に基づき、以下の **3つのソース** から修正内容を収集する。
3つの情報を突き合わせることで、漏れなく・正確に修正内容を把握する。

#### 2-1. 日付範囲の解釈

| 入力例 | 解釈 |
|--------|------|
| `4/1~` | 4/1（当年）から今日まで |
| `3/28~4/5` | 3/28〜4/5 |
| `2025/12/28~1/5` | 年をまたぐ場合は前半を前年と解釈 |

日付指定がない場合は **デフォルト動作** を適用する（後述）。

#### デフォルト動作（日付指定なし）

以下の2つを合わせて修正内容を把握する。

1. **現在のチャット（このコマンドを実行しているチャット）の修正内容**
2. **site-annotator の自動修正コミット**（存在する場合のみ）

site-annotator コミットの確認方法:

```bash
git log --oneline --all --grep="fix: AI自動修正 by site-annotator"
```

該当コミットが **1件以上ある場合は最新の1件のみ** を対象とし、その diff を取得して修正内容に含める。
該当コミットがない場合はチャットの修正内容のみで報告文を生成する。

#### ソース A: Git 履歴

日付指定ありの場合:

```bash
git log --oneline --after="{開始日}" --before="{終了日の翌日}"
```

対象コミットが特定できたら、範囲全体の diff を取得する。

```bash
git diff {最古のコミット}^..{最新のコミット}
```

デフォルト動作の場合は、上記の site-annotator コミットの diff のみを対象とする。

#### ソース B: エージェント履歴（agent transcripts）

agent transcripts フォルダのパスはシステムコンテキストに記載されている。
日付指定ありの場合は `ls -lt` でファイル更新日を確認し、指定期間内のものを対象とする。
デフォルト動作の場合は現在のチャットのみを対象とする。

各 `.jsonl` ファイルを行ごとにパースし、以下を抽出する。

**ファイル変更操作（tool_use）**

| ツール名 | 抽出する情報 |
|----------|-------------|
| `Write` | `path`（新規作成されたファイル） |
| `StrReplace` | `path` + `old_string` + `new_string`（何がどう変わったか） |
| `Delete` | `path`（削除されたファイル） |

**アシスタントの説明テキスト**

`role: "assistant"` かつ `type: "text"` のコンテンツから、何をしたかの説明文を取得する。
ツール呼び出しだけでは意図が分かりにくいため、説明テキストと合わせて修正の全体像を把握する。

#### ソース C: 対話履歴（ユーザーの指示）

同じ `.jsonl` ファイル内の `role: "user"` メッセージから、ユーザーがどんな修正を依頼したかを把握する。
修正の意図・背景を理解するために使用し、箇条書きの文言に反映する。

#### 除外対象

以下のパスへの変更は修正内容から除外する（内部ファイルのため）。

- `.cursor/` 配下のファイル
- `node_modules/` 配下
- `package-lock.json` / `pnpm-lock.yaml`
- `.next/` 配下

### STEP 3 — 修正内容を日本語に変換

抽出した変更操作とアシスタントの説明文を元に、以下のルールでクライアント向けの箇条書きに変換する。

#### 変換ルール

| やること | 具体例 |
|---------|--------|
| ファイル名・技術用語を使わない | `HomeCompanyInfo.tsx` → `会社情報セクション` |
| 同一変更は1行にまとめる | 4ファイルの営業時間変更 → `営業時間を10:00〜→9:00〜に変更` |
| 追加／削除／修正を明記 | `〜を追加` / `〜を削除` / `〜を修正・更新` |
| 場所が分かるように書く | `トップページの買取実績セクションに〜を追加` |
| CSSやレイアウト調整は具体的に | `余白の調整` ではなく `セクション間の間隔を広げて見やすく改善` |

#### 文体ルール（テンプレート全体に適用）

宛先のクライアント本人に直接送るメールとして自然な文体にする。

**禁止表現**

- 「先方」「先方様」「お客様側」など、宛先を外から指す三人称
- 社内用語がそのまま残っている言い回し
- 誰に向けた文か曖昧になる主語

**推奨表現**

| 避ける | 使う |
|--------|------|
| 先方からの依頼に基づき | ご依頼いただいたとおり |
| お客様側で指定された内容 | いただいたご指示に基づき |
| 先方の要望どおり | ご要望のとおり |
| クライアント確認後に | ご確認いただけましたら |

### STEP 4 — テンプレートに埋め込んで出力

以下のテンプレートに値を埋め込み、**完成した送信用本文をコードブロック（```）で囲んで出力する**。
ユーザーがそのままコピー＆ペーストできるようにする。
`{vercel_url}` と `{annotator_url}` にはSTEP 1で取得したURLを埋め込む。
取得できなかった場合はプレースホルダのまま出力する。
宛先名・差出人名はすべて `〇〇` のまま出力する（ユーザーが貼り付け後に手入力）。

テンプレート:

```
〇〇様

お世話になっております。
制作ディレクターの〇〇でございます。

ご依頼いただいておりました修正が完了いたしましたので、ご確認をお願いいたします。

▼サイトURL
{vercel_url}

▼修正内容
・{修正内容1}
・{修正内容2}
・{修正内容3}

▼修正ツール（追加の修正がある場合はこちらからご指示ください）
{annotator_url}

PC版・スマートフォン版ともにご確認いただき、問題なければドメイン接続の手続きに進めさせていただきます。
追加の修正点がございましたら、上記の修正ツールからお申し付けください。

メール受信から3営業日以内にご確認・修正依頼をいただけますと幸いです。

何卒よろしくお願いいたします。
```

## 出力後のルール

- 修正内容の箇条書きが **3件未満** の場合はそのまま出力
- **多い場合** は関連するものをまとめて **5件以内** に収める
- **多すぎる場合**（大量の変更で意味のある要約が困難な場合）は `▼修正内容` ブロック自体を削除する
- 変換した箇条書きの内容が不自然な場合は、出力後にユーザーへ「修正しますか？」と確認する
